## 2. New Issues Found Today (introduced since 2026-04-10)

The recent beta-access gating and admin work introduced 4 small issues. All fixed in this pass.

### NEW-01 — `BetaGate` flashes protected content during profile load
**Severity:** Medium (UX/privacy)
**File:** `src/components/BetaGate.tsx`

Original code returned `children` while `loading === true`. For a logged-in but unapproved user, the dashboard would briefly render before the profile arrived and BetaGate flipped to the waitlist screen. Real flash, even if short.

**Fix:** Branches reordered. While `loading` is true OR `user` exists but `profile` hasn't loaded yet, show a neutral spinner instead of `children`. Only the unauthenticated branch (`!loading && !user`) passes through to let pages handle `/login` redirect.

### NEW-02 — Admin email hardcoded in two places
**Severity:** Low (maintainability)
**Files:** `src/app/api/admin/approve-beta/route.ts`, `src/app/api/admin/pending-users/route.ts`

`const ADMIN_EMAILS = ['angelo.maniraguha@gmail.com']` was duplicated in both routes. Adding an admin meant editing two files.

**Fix:** Created `requireAdmin(request)` in `src/lib/auth.ts`. Reads `ADMIN_EMAILS` env var (comma-separated) with the founder's email as fallback. Both admin routes now call `requireAdmin` and the duplication is gone.

### NEW-03 — `pending-users` admin route missing `.limit()`
**Severity:** Low (defensive)
**File:** `src/app/api/admin/pending-users/route.ts`

Profiles query had no LIMIT. With 100s+ users, this returns the entire table to the admin client.

**Fix:** Added `.limit(1000)` cap. Plenty of headroom for the 100-tester beta and protects against runaway growth.

### NEW-04 — `store-rankings` query cap was 30k rows × 2
**Severity:** Low (defensive)
**File:** `src/app/api/store-rankings/route.ts`

Public marketing endpoint was capped at 30,000 rows per source (60,000 total). Already rate-limited at 10/min, but each request still pulled a giant payload.

**Fix:** Reduced cap to 10,000 per source. Sufficient for chain-level aggregation and a meaningful reduction in egress + memory.

## 3. Items Confirmed As Intentional Design (Not Fixing)

- **HIGH-07:** Rate-limit fail-open during Redis outage. The comment in `rate-limit.ts:124-128` explicitly chooses availability over strict enforcement. A Redis outage is rare; locking out all users during one is worse than the abuse window.
- **CRIT-05 partial:** `store-rankings` is intentionally unauthenticated (marketing/landing page consumer). With rate limiting and the reduced 10k cap, the residual risk is acceptable.
- **HIGH-03 cosmetic:** The legacy `normaliseProductName` (British spelling) in `sync-market-prices/route.ts` is functionally equivalent to the lib version for the only inputs that exist in the wild (French receipt data, no ligatures). Replacing it would change the normalization key for any historical row containing `œ`/`æ` and risk creating duplicate price-stat entries. Better to leave the existing key stable.
